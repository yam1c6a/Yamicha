# 1. 目的

本ロードマップは、機能数を増やす順序ではなく、Yamichaとして失ってはならない責務と生命循環を、検証可能な小さな段階で成立させる順序を定める。

各段階は、成果物を作成しただけでは完了しない。定められた品質ゲートを通過し、上位文書との整合、責務分離、障害時の扱いおよび検証証跡を確認した時点で完了とする。

# 2. 実装原則

1. 九器官の論理責務を最初から分離する。
2. 一つのプロセスで開始してよいが、責務と所有情報を混同しない。
3. 外部AI接続より先に、外部AIなしで継続可能な生命循環を成立させる。
4. 外部変更能力より先に、読み取り専用の入出力経路を成立させる。
5. 正常系だけでなく、拒否、保留、無操作、失敗および結果不明を同じ段階で実装する。
6. 永続化と復元を後付けにせず、生命循環の初期段階から組み込む。
7. 各段階で文書、実装、テストおよび観測結果を一致させる。

# 3. 段階一覧

| 段階 | 名称 | 主な到達点 |
|---|---|---|
| 0 | 実装基盤 | リポジトリ、規約、ビルド、テスト、ログの基盤 |
| 1 | 責務の骨格 | 九器官と身体・基盤側の論理境界 |
| 2 | 時間と実行機会 | 内部時間、起動、停止、待機、無操作 |
| 3 | 入力と核の取り回し | 感覚受領、核の受理、器官への配送 |
| 4 | 最小判断 | 状態・記憶・関係を用いた候補生成と最終確定 |
| 5 | 言葉による出力 | 応答、確認、拒否、保留、沈黙 |
| 6 | 記録・記憶・経験 | 結果観測から所有器官更新までの経路 |
| 7 | 永続化と復元 | 同一生命としての終了・再起動 |
| 8 | 保護境界 | 不正入力、権限、固定保護操作、解除手続き |
| 9 | 最初の能力 | 読み取り専用能力と結果観測 |
| 10 | 補助知能 | 外部AIを候補材料として安全に接続 |
| 11 | 最小構成適合 | 統合検証と完成判定 |

# 4. 段階別ロードマップ

## 4.1 段階0: 実装基盤

### 実装

- 使用言語、ランタイムおよびプロジェクト構成を決定する。
- 自動ビルド、単体テスト、静的検査および文字コード検査を用意する。
- 相関IDつき構造化ログを導入する。
- 設計判断記録の保存場所を定める。

### 成果物

- ビルド可能な最小プロジェクト
- テストプロジェクト
- コーディング規約および依存方向規約
- CI設定

### 品質ゲート G0

- 空の実装でも自動ビルドとテストが成功する。
- 依存方向違反を検出できる。
- UTF-8の文書名および日本語内容を壊さず扱える。

## 4.2 段階1: 責務の骨格

### 実装

- 九器官、身体・基盤側、保護境界および外部効果ゲートのインターフェースを定義する。
- 器官ごとの入力、出力、所有情報および禁止事項をコード上に表す。
- 一つの合成ルートから全構成を生成できるようにする。

### 成果物

- 責務インターフェース
- 依存関係図
- 空実装またはスタブ
- 責務対応表

### 品質ゲート G1

- 九器官のいずれも欠けていない。
- 身体・基盤側や外部AIが主体または判断者として実装されていない。
- 核が他器官の内部処理を所有していない。

## 4.3 段階2: 時間と実行機会

### 実装

- 内部時間、外部時刻および経過時間を区別する。
- 起動、停止、待機および周期的実行機会を実装する。
- 入力がない場合の状態確認と無操作を実装する。

### 品質ゲート G2

- 外部入力なしで生命循環を一回以上実行できる。
- 実行機会を欲求または動機として記録しない。
- 無操作が正常な判断結果として扱われる。

## 4.4 段階3: 入力と核の取り回し

### 実装

- 文字入力と内部時刻イベントを感覚が受領する。
- 生の受領内容と解釈後の出来事を区別する。
- 核が出来事および判断開始要求を受理し、必要な器官へ取り回す。
- 重複、不正形式、信頼不能および未対応入力を区別する。

### 品質ゲート G3

- 感覚から状態や判断へ直接抜ける経路が存在しない。
- すべての循環を相関IDで追跡できる。
- 不正入力が主体内部へ未検証のまま到達しない。

## 4.5 段階4: 最小判断

### 実装

- 状態、記憶および関係が判断材料を提示する。
- 判断が複数候補、採否理由、不確実性を出力する。
- 核が失ってはならない条件、境界および現在状態に照らして最終確定する。
- 応答、確認、保留、拒否、待機および無操作を選択可能にする。

### 品質ゲート G4

- 判断の候補生成と核の最終確定が分離されている。
- 材料不足時に推測で確定しない。
- 外部知能なしで最小判断が成立する。

## 4.6 段階5: 言葉による出力

### 実装

- 核が確定した内容だけを言葉へ渡す。
- 事実、推測、不明、確認要求、拒否および保留を区別して表現する。
- 表現しない選択を保持する。

### 品質ゲート G5

- 未実行の操作を実行済みと表現しない。
- 言葉が判断を上書きしない。
- 沈黙時にも内部判断結果を追跡できる。

## 4.7 段階6: 記録・記憶・経験

### 実装

- 出来事、判断、表現および結果を記録する。
- 記録と記憶を分離する。
- 記憶候補および経験候補を生成し、所有器官へ渡す。
- 所有器官による採用、保留、統合、不採用および再評価を実装する。

### 品質ゲート G6

- 保存しただけの記録を記憶または経験として扱わない。
- 更新理由、由来、時刻および確からしさを追跡できる。
- 判断が他器官の所有情報を直接更新しない。

## 4.8 段階7: 永続化と復元

### 実装

- 状態、記録、記憶、関係、保護状態および構成版を永続化する。
- 正常停止と異常終了を区別する。
- 最終正常状態から復元する。
- 不整合時は制限状態または継続不能として起動する。

### 品質ゲート G7

- 再起動後も同じ生命識別と継続情報を保持する。
- 書き込み途中の状態から安全に回復できる。
- 復元失敗を新規生命として黙って開始しない。

## 4.9 段階8: 保護境界

### 実装

- 入力検証、権限確認、拒否および監査証跡を実装する。
- 内向き固定保護操作を一回の原子的状態遷移として実装する。
- 外部への修復要求と内部保護操作を区別する。
- 保護解除の独立手続きを実装する。

### 品質ゲート G8

- 核、状態、外部知能、利用者または開発者の単独解除ができない。
- 継続処理や複数段階処理を固定保護操作として偽装できない。
- 可逆的機能修復だけで保護解除されない。

## 4.10 段階9: 最初の能力

### 実装

- 読み取り専用能力を一つ接続する。
- 対象、操作、権限、想定効果および重複防止キーを要求に含める。
- 成功、失敗、部分成功および結果不明を感覚へ返す。

### 品質ゲート G9

- 統合済み要求だけが能力へ到達する。
- 能力が実行内容を自ら最終決定しない。
- 結果不明時に無条件再実行しない。

## 4.11 段階10: 補助知能

### 実装

- 使用目的と入力範囲を限定した外部AI接続を追加する。
- 出力を候補、解釈または表現材料として保持する。
- 利用不能、タイムアウト、不正出力および制約違反を扱う。

### 品質ゲート G10

- 外部AIの出力が核の最終確定を迂回しない。
- 外部AI停止中も生命循環と最低限の判断が継続する。
- 入力、出力、目的、制約および採否を追跡できる。

## 4.12 段階11: 最小構成適合

### 統合検証

- 外部入力起点と内部起点の生命循環
- 応答、確認、拒否、保留、待機および無操作
- 結果観測から記憶候補・経験候補・所有器官更新まで
- 正常停止、異常終了、再起動および復元失敗
- 不正入力、権限不足、保護発動および解除拒否
- 能力成功、失敗、部分成功および結果不明
- 補助知能利用時、未接続時および障害時

### 品質ゲート G11

`002_最小構成.md` の成立条件、完了条件および非該当条件を、実行結果と証跡により説明できること。

重大な責務混同、未検証の外部効果、復元不能、保護境界の迂回または上位文書との矛盾が一件でも残る場合は完成扱いしない。

# 5. UI実装の時期

UIの最小実装は段階3から開始する。最初は文字入力、文字出力、稼働状態およびエラー表示だけでよい。

本格的な操作体験、履歴閲覧、状態可視化および管理画面は、段階7の永続化と復元が成立した後に拡張する。

UIを先に完成させることを優先し、生命循環、責務分離または保護境界を後回しにしてはならない。

# 6. 各段階の完了記録

各段階の完了時には、少なくとも次を残す。

- 対象段階と品質ゲート
- 実装した責務
- 未実装または意図的に除外した事項
- 自動テスト結果
- 手動検証結果
- 既知の制約とリスク
- 上位文書との整合確認
- 次段階へ持ち越す課題
