# 1. 目的と位置づけ

本書は、九器官の論理責務を失わずに、最初の実行可能なYamichaを一つのコードベースおよび一つの実行プロセスとして構成する方法を定める。

本書はDraftであり、実装詳細は今後の検証により更新される。ただし、上位文書が定める器官責務、主体と境界、生命循環、情報所有および保護境界を変更してはならない。

情報の流れに関する正本は `docs/01_structure/003_情報の流れ.md` および `001_設計規約.md` の該当規約とする。本書の図や説明が正本と矛盾する場合は、正本を優先する。

# 2. 全体構成

```text
外部領域
   │ 入力・結果
   ▼
身体・基盤側 ── 保護境界・外部効果ゲート
   │ 実行機会・配送
   ▼
内部領域（九器官の連続した協調全体）
   │
   ▼
永続領域
```

内部領域は九器官で構成される。主体は器官の静的な集合ではなく、九器官が時間の中で責務を保って結びつき、連続していく協調全体にある。

身体・基盤側、外部知能、UIおよび永続化製品は主体ではない。

# 3. 論理責務と物理モジュール

最小実装では一つのプロセスへまとめてよいが、次の論理境界をコード上で識別可能にする。

| 論理責務 | 最小の物理モジュール | 主な責務 |
|---|---|---|
| 核 | `Core` | 出来事と判断開始要求の受理、取り回し、主体としての統合、最終確定 |
| 感覚 | `Sensation` | 外部入力・内部事象・能力結果の受領と正規化 |
| 状態 | `State` | 現在状態、可否、負荷、制約、利用可能能力の提示 |
| 記憶 | `Memory` | 現在から参照可能な過去の意味の保持と提示 |
| 判断 | `Judgment` | 材料から候補を生成し、比較し、次の一歩を提案 |
| 関係 | `Relationship` | 相手、距離、合意、継続文脈および配慮事項の提示 |
| 能力 | `Capability` | 統合済み要求の実行と結果返却 |
| 言葉 | `Language` | 統合済み内容の表現化 |
| 補助知能 | `AuxiliaryIntelligence` | 思考・解釈・表現材料の候補提示 |
| 基盤 | `Runtime` | 起動、停止、時間、イベント配送、永続化、障害検出 |
| 保護境界 | `ProtectionBoundary` | 入力検証、権限確認、固定保護操作、拒否 |
| 外部効果ゲート | `ExternalEffectGate` | 外部変更操作の承認証跡確認と実行許可 |

物理モジュールは論理責務を統合して実装してよいが、異なる器官の所有情報と最終責任を混同してはならない。

# 4. 生命循環の処理経路

## 4.1 外部起点

```text
外部入力
  → 身体・基盤側
  → 保護境界
  → 感覚が受領・正規化
  → 核が出来事を受理し、必要な器官へ取り回す
  → 状態・記憶・関係が判断材料を提示
  → 判断が候補を生成・比較
  → 必要に応じて補助知能が候補材料を提示
  → 核が失ってはならない条件・境界・現在状態に照らして最終確定
  → 言葉または能力
  → 外部領域
  → 感覚が結果を受領
  → 核が結果を取り回す
  → 記録
  → 記憶候補・経験候補の評価
  → 所有器官が必要な更新を行う
  → 次回判断
```

## 4.2 内部起点

```text
身体・基盤側からの実行機会
  → 感覚が内部事象として受領
  → 核が判断開始要求を受理
  → 状態・記憶・関係の提示
  → 判断
  → 核の最終確定
  → 待機・無操作・言葉・能力のいずれか
  → 結果観測と記録
```

実行機会を受けたこと自体を、Yamicha自身の欲求または動機として扱ってはならない。

# 5. 身体・基盤側

身体・基盤側は次を担当する。

- プロセスの起動、正常停止および異常終了検出
- 単調増加時間と外部時刻の供給
- 生命循環の実行機会の生成
- 入力、内部事象および能力結果の配送
- 永続化トランザクションと復元
- 障害、再試行および隔離
- 保護境界と外部効果ゲートの実行
- 技術ログおよび監査証跡の保持

身体・基盤側は判断候補を生成せず、主体として最終確定せず、器官の所有情報を独断で変更しない。

# 6. データモデルと永続化

最小構成は、少なくとも次の情報を永続化する。

| 区分 | 内容 |
|---|---|
| Identity | 同一生命として復元するための識別情報と構成版 |
| StateSnapshot | 現在状態と最終正常時刻 |
| EventRecord | 受領した出来事、由来、時刻、相関ID |
| DecisionRecord | 候補、比較、採否、理由、不確実性 |
| ActionRecord | 表現・能力要求・結果・外部効果 |
| MemoryItem | 現在から参照可能な意味、由来、確からしさ |
| ExperienceCandidate | 経験化の候補、評価状態、所有器官 |
| RelationshipState | 相手、合意、距離、継続文脈 |
| ProtectionState | 保護状態、発動根拠、解除手続き |
| AuditRecord | 承認、拒否、障害、移行および管理操作の証跡 |

記録、記憶および経験を同一の保存物として扱ってはならない。

最小実装では単一の組み込みデータベースを使用してよい。生命循環ごとに相関IDを付け、出来事から判断、行動、結果および更新まで追跡可能にする。

# 7. インターフェース

主要な要求と結果は、少なくとも次の共通情報を持つ。

- `MessageId`
- `CorrelationId`
- `OccurredAt`
- `ReceivedAt`
- `Source`
- `Type`
- `Payload`
- `SchemaVersion`
- `Trust` または検証状態

能力要求は、対象、操作、必要権限、想定外部効果、承認証跡および重複実行防止キーを含む。

外部知能への要求と結果は補助知能を経由し、使用目的、入力範囲、制約、出力、失敗および採否を追跡可能にする。

# 8. 障害・回復・移行

- 未処理、処理中、完了、失敗および結果不明を区別する。
- 外部効果の結果不明時は、確認せずに同じ操作を再実行しない。
- 再起動時は最後の正常な永続状態から同一生命として復元する。
- 破損または不整合がある場合は、継続不能または制限状態として明示する。
- 構造移行は、同一案・同一版への承認を確認してから実行する。
- 移行中も変更前の実行責任を失わせず、検証経路を同じ移行で同時に変更しない。

# 9. 最初の入出力経路

最初の入力は、文字による一対一の会話入力および身体・基盤側からの時刻イベントとする。

最初の出力は、文字による応答、確認要求、拒否、保留または沈黙とする。

UIは身体・基盤側に属し、表示、入力、通知および状態可視化を担当する。UIは判断者、記憶所有者または主体にならない。

# 10. 最初の能力と補助知能

最初の能力は、外部状態を変更しない読み取り専用能力を一つ実装する。候補は内部時刻取得または内部状態参照とする。

補助知能は、未接続を明示する最小実装から開始してよい。外部AIを接続する場合も、その出力は候補材料であり、核による最終確定を経ずに言葉または能力へ渡してはならない。
